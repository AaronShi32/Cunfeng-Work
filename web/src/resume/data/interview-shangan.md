## 四、岗位认知类

### Q1：你是企业出身，为什么想来自然资源部信息中心？能适应吗？

**思路：认同价值 + 能力迁移 + 姿态谦逊。**

我在企业积累的是「大规模数据平台 + AI 落地」的硬能力，但企业追求的是商业效率；而自然资源信息化承载的是**耕地保护、国土空间规划、生态安全**这些国家级公共价值，数据的公共属性和合规要求更强，这种「技术服务于国计民生」的价值感是我特别看重的。

能力上我并不担心适应：时空数据本质仍是海量异构数据治理，遥感智能应用本质仍是 AI 工程落地，这些我都有实战。我需要补的是**行业知识**——国土空间平台的数据标准、调查监测的业务规则、政务信息化的合规规范。这些是「学得会」的领域知识，我已经在系统学习相关政策文件；而工程与 AI 的底层能力是「练出来」的，正是我能立刻贡献的部分。

### Q2：数据与 AI 类岗位，你觉得最重要的能力是什么？

我认为是**「让 AI 在守底线的前提下真正解决业务问题」的落地能力**，具体三层：

1. **数据治理是地基**：多源时空数据的标准化、质量、时空基准统一，数据不干净，模型再好也没用。
2. **模型 / 算法服务业务**：遥感变化检测、图斑智能提取、智能问答，都要嵌入真实业务闭环（如耕地保护监测），而不是炫技的 demo。
3. **合规与工程化是护栏**：涉密分级、等保、可观测、可回溯。我在 Scout 项目里最大的体会就是——AI 输出必须**可溯源、可解释、人可复核**，尤其在政务场景，不能让模型「拍脑袋」。

## 五、综合分析类

### Q3：有人说「有了大模型，遥感解译和数据处理都能全自动化」，你怎么看？

**破题：肯定价值 + 指出边界 + 落到实践。**

- **肯定**：大模型和深度学习确实极大提升了遥感变化检测、图斑提取、智能问答的效率，把人从重复标注中解放出来，这是趋势。
- **但要辩证看边界**：
  - **数据质量决定上限**：多源异构、时相不一致、云遮挡等问题，模型无法凭空补齐，前置的数据治理不可替代。
  - **精度与责任**：耕地「非农化」认定、违法用地判定关系到执法和考核，误判代价极高，必须**人机协同、人工复核终审**。
  - **合规红线**：涉密地理数据不能随意喂给外部大模型，要走**私有化部署 + 数据脱敏 + 分级管理**。
- **落到实践**：正确姿态是「**AI 辅助 + 专家把关**」——我在 Scout 里就是让 AI 做初筛和报告生成，最终结论由工程师确认。同样，遥感智能解译应定位为「提效工具」，而非「替代决策」。

**收尾**：技术要拥抱，但在自然资源这种严肃业务场景，越是先进的工具，越要配套治理机制和责任闭环。

### Q4：谈谈你对「数据要素」在自然资源领域价值的理解。

自然资源数据（国土、地籍、遥感、规划）是典型的**高价值公共数据要素**。价值释放的关键是「三个统一 + 一个安全」：统一时空基准、统一分类标准、统一共享平台（即国土空间基础信息平台），在**安全合规分级**前提下，让数据在部门间、政企间有序流动，支撑规划、监测、执法、决策。我做过跨系统数据打通和共享服务治理，深知**「共享的前提是标准和安全」**——这也是自然资源数据要素化最需要啃的硬骨头。

## 六、应急 / 危机处理类

### Q5：国土空间基础信息平台在一次重要汇报前突发大面积卡顿 / 无法访问，你作为技术人员如何处理？

**结构：稳（先止血）→ 定（快定位）→ 通（先恢复）→ 复（复盘）。**

1. **稳住 + 分级**：立即启动应急响应，第一时间同步给负责人和汇报方，给出预期；明确影响范围（全量 or 局部、只读 or 全挂）。
2. **快速定位**：按依赖链排查——网络 / 网关 → 应用服务 → 数据库 / 时空数据服务 → 底层算力。我在云平台做稳定性治理时依赖**可观测性三板斧（日志、指标、链路）**先看监控大盘定位瓶颈层，而不是盲目重启。
3. **优先恢复**：以「先恢复业务、后彻底修复」为原则——限流降级、切换备用节点、只读兜底、缓存/静态一张图应急展示，确保汇报能进行。
4. **闭环复盘**：事后做无指责复盘（root cause + 改进项），补齐监控告警、容量评估、应急预案，避免复发。

**升华**：这类平台是政务级系统，我会推动建立**常态化健康巡检 + 应急预案演练**机制——我在阿里云就是靠「定时巡检 + 健康诊断 + 事件通知」把大规模迁移的成功率和稳定性撑起来的，同一套方法论完全适用。

## 七、人际沟通 / 协作类

### Q6：业务处室（如耕地保护）要求两周内上线一个遥感监测智能预警功能，但你评估数据治理和模型验证至少需要一个月，如何沟通？

**思路：不硬顶、不盲从，用专业换信任。**

1. **先理解诉求**：业务方的急，往往背后有真实的督查 / 考核压力，先听清「为什么急、最核心要什么」。
2. **拆解 + 给方案**：不说「做不了」，而是拆成**分阶段交付**——第一周先上「重点区域 + 人工复核」的最小可用版本满足急需，第二阶段再补全数据治理和精度验证。
3. **讲清风险**：明确告诉对方，跳过数据治理和验证会带来**误报漏报和执法风险**，用业务方能听懂的代价来对齐，而不是技术术语。
4. **达成一致 + 留痕**：确定分期计划和责任边界，形成书面共识。

**收尾**：跨团队协作我做了很多年，核心是**「用对方的语言讲清取舍，让专业判断成为大家的共同决策」**，既守住质量底线，也照顾业务紧迫性。

<details>
<summary>Q：涉密数据和公有云 / 大模型怎么平衡？</summary>

原则是**「涉密不出域」**：涉密地理数据走**私有化 / 专网部署**，大模型采用本地化部署或私有化推理，绝不调用外部公有 API；对需要 AI 处理的数据做**分级 + 脱敏**，公开级才可用更灵活的方案。我在微软做安全合规迭代（证书轮转、合规变更）时，核心就是「先定密级和权限边界，再谈能力」，这套习惯直接适用。
</details>

<details>
<summary>Q：Scout 这个 AI 项目具体怎么落地的？</summary>

Scout 是把三块拼起来：① **LLM** 做理解和生成；② **MCP 检索引擎** 做实时日志 / 数据检索（相当于给模型接上「眼睛」）；③ **Wiki 知识库** 提供领域知识做检索增强（RAG）。流程是：用户描述问题 → Agent 检索相关日志和知识 → LLM 综合分析 → 生成诊断报告，工程师复核。映射到自然资源，就是「政策文件 + 业务数据 + 大模型」的智能问答 / 辅助决策，技术路线完全相通。
</details>

<details>
<summary>Q：为什么频繁强调「合规底线」？会不会拖慢创新？</summary>

在政务和自然资源场景，合规不是创新的对立面，而是创新能否上线的前提——一个不合规的智能应用，做得再好也上不了线、担不了责。真正的工程能力是**「在合规约束内把事做成」**。我做云平台稳定性就是在「不能出故障」的强约束下持续优化，反而逼出了更扎实的方案。
</details>
