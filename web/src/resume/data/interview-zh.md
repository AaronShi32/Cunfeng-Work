# 技术难点

## 1. HDI 网络模型：四种接入模式

> 体现 PaaS 如何通过灵活的网络设计满足不同企业客户的 IaaS 级要求。

### 背景

Azure HDInsight 作为托管的大数据 PaaS，需要在客户的 Azure 订阅内部署集群节点。不同客户对网络隔离、合规、出口流量管控有不同要求，因此形成了四种网络模式。

### 四种模式对比

| 模式 | 方向 | 核心特征 |
|---|---|---|
| **Inbound** | 外部 → 集群 | 客户通过公网 IP 或 Private Endpoint 访问集群服务（Ambari、Jupyter、REST API）|
| **Outbound** | 集群 → 外部 | 集群访问 Azure 服务（Storage、Key Vault、Azure Monitor）及外部依赖（Maven、PyPI）|
| **ESP（Enterprise Security Package）** | 身份集成 | 通过 Azure AD DS 或 LDAP 打通企业身份，实现 Kerberos 认证 + Ranger 鉴权 |
| **BYO VNet（Bring Your Own VNet）** | 网络自托管 | 客户提供自己的子网，HDI 在客户 VNet 内部署节点，完全控制路由、NSG、防火墙规则 |

### 技术难点分析

**Inbound 难点**：需要维护一份动态的 HDI 管理 IP 白名单（约 60+ 个地区 IP），并通过 NSG 规则或 Service Tag 放行，同时支持 Private Endpoint 方案满足零公网场景。

**Outbound 难点**：集群依赖的外部端点（FQDN）超过 100 个，分散在 Azure 服务和第三方镜像源。客户强制出口流量走防火墙时，需要提供完整的 FQDN 白名单；若漏配则导致集群创建失败，排查成本极高。

**ESP 难点**：Kerberos 票据有时效性，节点时钟同步（NTP）、AD 连通性、DNS 解析三者缺一不可。MSI 凭据轮换、KDC 连接失败、Ranger 策略同步延迟是最常见的故障根因。

**BYO VNet 难点**：HDI 依赖子网内的特定路由（UDR 不能屏蔽对 Azure 管理平面的访问）、特定 DNS（需能解析 Azure 内部域名），以及子网空间要足够大。客户自带 VNet 意味着 HDI 对网络环境几乎没有控制权，排查问题时需要深入客户侧配置。

### 面试角度

> 这四种模式本质上是 PaaS 服务适配 IaaS 客户需求的典型解法：通过可配置的网络边界，把「合规、隔离、身份集成」这些企业级需求叠加到托管服务上，而不是让客户裁剪业务来适应平台限制。

<details>
<summary>面试建议回答（一句话版）</summary>

HDI 的四种网络模式对应不同的企业安全诉求：Inbound 控入口访问，Outbound 管出口依赖，ESP 打通企业身份，BYO VNet 让客户完全掌控网络边界。核心挑战是如何在不控制客户环境的情况下，保证平台依赖链路完整可达。

</details>

## 2. FCS 启动速度优化：ACI → Image Cache → Pool

> 从 10 分钟级冷启动到分钟内就绪，三代优化方案的演进路径。

### 背景

FCS（Fabric Container Service）负责管理 Microsoft Fabric 的计算工作负载，每次用户提交 Spark/ML 任务，都需要启动一批容器。容器冷启动延迟直接影响用户体验，也是与竞品对比的核心指标之一。

### 三个优化阶段

**阶段一：基础 ACI 模式（冷启动）**

- 每次任务触发时，FCS 向 ACI 发出创建请求
- 镜像从 ACR 拉取（2–5 GB），节点注册，服务就绪
- 端到端延迟：**8–12 分钟**
- 问题：镜像拉取是主要瓶颈，约占 60% 启动时间

**阶段二：Image Cache（镜像预热）**

- 在 AKS 节点池上预先拉取镜像（DaemonSet 模式），节点维持「镜像已就绪」状态
- 任务下发时跳过拉取阶段，直接创建容器
- 端到端延迟降至：**3–5 分钟**
- 问题：节点仍需分配和初始化，有调度排队窗口；镜像版本升级时需要滚动预热，有短暂延迟窗口

**阶段三：Container Pool（容器预池化）**

- 后台保持一批「空转容器」处于 Ready 状态，等待工作负载注入
- 任务到来时直接从 Pool 分配容器，传入用户代码和配置，而非重新创建
- 端到端延迟降至：**< 60 秒**（甚至秒级）
- 设计挑战：
  - Pool 容量如何预测（过大浪费资源，过小无法覆盖峰值）
  - 容器健康检查与自动补充（Container 可能因为依赖失败而 Crash）
  - 安全隔离：不同租户的容器不能共享 Pool，需按规格/SKU 分桶
  - Pool 中容器的身份（MSI）、挂载（Secret/Config）需要在分配时动态注入

### 关键权衡

| 方案 | 启动延迟 | 资源成本 | 运维复杂度 |
|---|---|---|---|
| 冷 ACI | 8–12 min | 低（按需计费）| 低 |
| Image Cache | 3–5 min | 中（节点常备）| 中 |
| Container Pool | < 60 s | 高（容器预占）| 高 |

> 实际选择取决于 SLA 要求和预算。Pool 方案适合高频、低延迟诉求；冷 ACI 适合低频、对成本敏感的场景。

<details>
<summary>面试建议回答（一句话版）</summary>

FCS 的启动优化经历了三代：从 ACI 冷拉镜像（10 分钟），到 Image Cache 跳过拉取（3–5 分钟），再到 Container Pool 预分配就绪容器（< 60 秒）。核心是用「预热换延迟」，代价是资源成本和运维复杂度的提升，选哪个取决于 SLA 目标。

</details>

## 3. Scout Agent 诊断稳定性

> 让 AI 诊断工具在生产环境中真正可信、可用，而不是只能演示。

### 背景

Scout 是 FCS / HDInsight 平台内部的智能诊断 Agent，结合 LLM + 日志检索引擎 + Wiki 知识库，帮助 On-call 工程师快速定位线上问题。核心挑战：AI 生成的诊断结论如何做到可信、可复现、可审计。

### 主要稳定性挑战

**1. 幻觉与误诊**

LLM 在上下文不足时可能「编造」根因或给出无依据的修复建议。

解决方式：
- 强制要求每条诊断结论附带「来源引用」（日志行号、Wiki 链接）
- 设置「置信度阈值」，低于阈值时输出「无法确定，建议人工介入」而非给出错误结论
- 诊断链路可视化：展示「调用了哪些工具、检索了哪些数据、得出结论的推理步骤」

**2. 工具调用可靠性**

Scout 通过 Function Calling 调用日志检索、Kusto 查询、ITSM 工单等外部工具，任何工具超时或返回空结果都可能导致链路断裂。

解决方式：
- 每个工具调用设置 timeout + retry，超时时返回「检索失败」而非静默跳过
- 关键工具（如日志检索）设置 fallback：主链路失败时切换到备用数据源
- 工具返回结果做类型校验，防止 LLM 把错误格式的响应当作有效数据

**3. 上下文窗口管理**

一次线上故障涉及的日志量可能达到 MB 级，远超 LLM 上下文长度。

解决方式：
- 检索阶段先用关键词过滤 + 时间窗口裁剪，只取最相关的 N 条日志
- 分阶段诊断：先宽泛检索找到异常时间点，再精细检索该时间段的详细日志
- 避免把原始日志全量塞入 prompt，改为「摘要 + 关键片段」的结构化输入

**4. 评估与持续迭代**

如何知道 Scout 诊断得好不好？没有评估体系等于盲飞。

解决方式：
- 建立 Golden Set：收集历史 Incident 的「问题 → 已验证根因」对，作为评估基准
- 定期跑回归：新版本 Scout 跑 Golden Set，对比准确率和漏诊率
- On-call 反馈闭环：诊断结论被采纳/驳回时记录，反哺模型微调或 prompt 优化

### 面试角度

> Scout 的核心不是「让 LLM 看日志」，而是「构建一套让 LLM 的输出可信、可验证、可迭代的工程体系」。稳定性来自：工具调用有防护、输出有根据、失败有降级、效果有度量。

<details>
<summary>面试建议回答（一句话版）</summary>

Scout 的诊断稳定性靠四层保障：工具层的 timeout + fallback、LLM 层的来源引用 + 置信度阈值、上下文层的检索裁剪 + 结构化输入、评估层的 Golden Set + 反馈闭环。目标是让 AI 的结论「可信可审计」，而不是「看起来合理但无法验证」。

</details>
