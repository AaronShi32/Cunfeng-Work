## 项目一：Microsoft Fabric / Azure HDInsight 容器基础设施平台

### 可引导面试官追问的点

1. **ACI + AKS 选型与职责边界**：为什么是这两类底座组合？分别承载什么工作负载？
2. **控制面 / 数据面设计**：Workload 生命周期如何拆分，RESTful API 如何抽象，如何保证可扩展性？
3. **容器化降本 80%**：成本模型如何测算，资源利用率和性能损耗如何权衡？
4. **Telemetry 与稳定性体系**：日志、指标、链路、发布治理如何打通？
5. **Scout 智能诊断 Agent**：LLM、日志检索引擎、Wiki 知识库如何协同，如何控制幻觉与误诊？
6. **On-call 与退役阶段治理**：如何在存量客户阶段兼顾稳定性、安全性和变更风险？

### 建议重点准备的问题

- 请详细介绍 FCS 的控制面和数据面架构，以及它们之间的通信机制。
- 从 VM 迁移到 Container 带来 80% 成本下降，这个结果是如何测算和验证的？
- 你设计的 Telemetry 架构包括哪些核心数据面，如何支撑问题定位和发布治理？
- Scout Agent 的诊断链路是什么？如何选择工具、组织上下文并输出可信报告？
- 服务可用性长期保持在 99%+，你做过哪些关键稳定性改造？

## 项目二：阿里云 ECS 弹性网卡（ENI）与跨可用区实例迁移

### 可引导面试官追问的点

1. **ENI 核心能力设计**：多私网 IP、IPv6、eRDMA 分别带来哪些平台复杂度？
2. **事件驱动重构**：为什么要从轮询改成事件驱动？消息可靠性、顺序性、幂等性怎么保证？
3. **接口幂等设计**：请求重试、并发冲突、状态回放如何处理？
4. **深分页与慢 SQL 优化**：NextToken 为什么优于 Offset，索引如何设计到 3ms？
5. **跨可用区迁移保留私网 IP**：难点在哪里，为什么这是高价值能力？
6. **60 万台实例迁移**：如何灰度、回滚、监控和自动化运维？

### 建议重点准备的问题

- 请举一个具体 API，说明你是如何做接口幂等设计的。
- 事件驱动重构中，如何处理重复事件、乱序事件和回调超时？
- 百万级查询从 300ms 到 3ms，SQL 和索引分别做了什么优化？
- 跨可用区迁移中，为什么“迁移后私网 IP 不变”难做？你的方案核心是什么？
- 60 万台实例迁移如何分批推进，如何保证迁移过程对业务影响可控？

# 面试亮点问答版

> 基于简历内容与 HDInsight Wiki 中可对应的设计、TSG、On-call、Autoscale、证书/身份、网络依赖、Fabric 发布流程等材料整理。

## 1. 控制面 / 平台工程能力

**简历亮点：** 长期负责云平台控制面、容器基础设施、网络与稳定性治理，不是单点业务开发。

**提问：** 你在微软这段经历里，最能体现平台架构能力的事情是什么？

**回答：**  
我做的工作更偏控制面和平台治理，而不是单点功能开发。比较典型的是参与 HDInsight / Fabric 相关平台能力建设，覆盖工作负载接入、发布流程、运维、证书、网络依赖和故障处理。  
这类工作的核心不是写一个接口，而是把接口设计、依赖治理、上线策略、异常恢复和跨团队协作串起来。所以我的经验更接近平台工程和基础架构，而不是纯应用层开发。

---

## 2. 稳定性治理 / On-call / RCA

**简历亮点：** 不只是处理线上问题，更能把 incident 处理流程、TSG、repair item、postmortem 做成闭环。

**提问：** 你做稳定性提升时，通常怎么落地？

**回答：**  
我会把稳定性治理拆成四层：  
第一层是 on-call 流程标准化，比如告警确认、影响面判断、是否 request assistance、如何交接。  
第二层是 TSG 和 RCA 沉淀，把一次 incident 的分析过程固化成下次可复用的手册。  
第三层是可观测性建设，把日志、指标、依赖链、发布信息串起来。  
第四层才是根因修复，把 repair item 纳入后续版本，而不是一次性救火。  
我更关注的是“检测、定位、缓解、复盘、预防”的完整闭环。

---

## 3. Telemetry / 可观测性建设

**简历亮点：** 具备从日志采集到数据集、任务依赖、监控看板、失败恢复的体系化思维。

**提问：** 你做 Telemetry 时，怎么避免只停留在“打日志”层面？

**回答：**  
我会先反推谁消费这些数据、要拿它做什么决策。  
在 HDInsight Telemetry 这类场景里，不只是把 Geneva/Kusto 日志收上来，还要沉到 Cosmos、SQL、Lens、Power BI，形成数据集、依赖关系和监控视图。  
我会重点关注四件事：数据源分层、处理链路可追踪、失败可恢复、最终能支持业务分析和运维决策。  
所以我做 Telemetry 的目标不是“有日志”，而是“能定位问题、能驱动运营、能支持发布决策”。

---

## 4. Autoscale / Workflow / 性能与可靠性平衡

**简历亮点：** 能把自动扩缩容讲成完整系统，理解消息链路、workflow、feature flag、时延优化和故障面治理。

**提问：** 你怎么理解 Autoscale 这类能力的技术难点？

**回答：**  
Autoscale 难的不是多扩几台机器，而是如何在低延迟、可靠性和业务无损之间做平衡。  
比如 Autoscale v2 的目标就是把 P90 scale latency 压到 12 分钟以内，同时提升可靠性。它背后是一整条链路：集群侧生成请求、用密钥加密、通过 queue 投递、RP 侧解密并启动 workflow，再配合多个 feature flag 缩短等待和减少阻塞。  
我做这类系统时，会重点关注端到端时延和阻塞点治理，而不只是某一个 workflow 的实现。

---

## 5. 分布式系统运维细节理解

**简历亮点：** 能深入到节点注册、脚本执行、调度约束、状态机时序，不只是看 dashboard。

**提问：** 你怎么证明自己是真的理解系统内部机制？

**回答：**  
我能直接讲到节点级机制。比如扩容时新 worker node 刚注册到 Ambari，如果 persisted script action 还没跑完，YARN 过早把任务调度上去就会出问题。  
所以需要先打上 node label，比如 `initializing`，等脚本执行完再移除。  
这说明很多平台故障，本质上是多个子系统之间的时序和状态机问题，而不是某个接口单点报错。  
我比较擅长把这类“表面现象”拆回到底层机制。

---

## 6. 安全 / 证书 / MSI / 密钥轮换

**简历亮点：** 有生产级证书链迁移、IPSec 证书轮换验证、MSI renew 故障、凭据泄漏处置等经验。

**提问：** 你做过哪些偏安全和身份链路的工作？

**回答：**  
这块我做过比较多偏生产环境的问题处理，包括 MSPKI CA Migration、OneCertV2 PrivateCA 下的 IPSec 证书验证、RenewMsiCredentialWorkflow 失败分析，以及 Credential Leak 处置。  
我对这类问题的理解不只是安全概念，而是工程落地：证书如何发放、如何轮换、如何验证链路是否生效、问题发生后怎么做回滚和恢复。  
所以我在安全问题上更偏“可落地、可运维、可恢复”的视角。

---

## 7. 复杂故障定位能力

**简历亮点：** 能把现象、日志、身份/证书、数据库、KeyVault、恢复动作串成完整根因链路。

**提问：** 能讲一个你定位根因比较深入的案例吗？

**回答：**  
可以讲 MSI renew workflow failed due to db encryption cert missing 这一类案例。  
表面现象是 workflow timeout，但往下查会发现 gateway 节点无法加载证书，再往下能看到 manifest signing context 解密失败。  
继续分析后发现 RP 侧缺失 DB encryption cert，导致发给 cluster 的 signing cert 不对，最终 cluster 更新 manifest 失败，并连带 MSI renew 失败。  
我处理这类问题时，不会停在单条报错上，而是把现象、日志、证书链路、DB 状态、KeyVault、修复动作和回归验证整条链打通。

---

## 8. 网络依赖治理 / 客户环境适配

**简历亮点：** 具备把外部依赖显式化、程序化发现和面向客户网络边界治理的思维。

**提问：** 你做平台时，怎么处理网络相关问题？

**回答：**  
我比较重视把依赖显式化，而不是让客户自己猜。  
比如 FQDN 这类工作，本质上是在定义系统到底依赖哪些外部端点，包括登录、监控、Service Bus、Queue、证书下载等。  
这样网络问题就不再是模糊判断，而是可以变成配置清单和接口契约。  
我一般会把网络问题拆成两类：系统自身依赖是否定义清楚，以及客户环境是否满足这些依赖。

---

## 9. 发布体系 / DevOps / 环境治理

**简历亮点：** 熟悉从 repo、build、package、train、测试环境到部署看板的整套发布链路。

**提问：** 你做过发布或 DevOps 体系的事情吗？

**回答：**  
做过，而且不是只会点 pipeline。  
像 Fabric 这类场景里，repo、official build、PR pipeline、package update、release train、CST/Edog/Daily/Prod、deployment dashboard 这些环节是串起来的。  
这说明我不只是写功能代码，还理解版本如何进入不同环境、怎么做 validation、如何跟 train 节奏对齐、怎么观察部署状态和处理环境差异。  
很多线上问题和版本节奏、包依赖、环境差异、回滚策略强相关，所以平台工程师必须懂发布。

## 最值得优先主打的 5 个面试点

1. **控制面 / 平台工程能力**：最能体现你和普通后端的差异。  
2. **稳定性治理 / On-call / RCA**：体现 owner 意识和生产经验。  
3. **Autoscale / Workflow / 调度链路**：适合讲复杂分布式系统设计与故障处理。  
4. **安全证书 / MSI / 密钥轮换**：稀缺，且有很强生产实战感。  
5. **发布体系 / DevOps / 环境治理**：能把你拔高到平台负责人/骨干画像。

## 使用建议

- 面试时优先讲你**直接做过且证据最强**的方向：控制面、稳定性、Autoscale、安全证书/MSI。  
- AI / Scout / LLM 可以讲，但建议表述成**参与建设和落地**，不要讲成绝对主导。  
- 每个回答尽量遵循 **背景 -> 难点 -> 方案 -> 结果 -> 复盘** 的结构。
